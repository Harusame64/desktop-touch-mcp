  Opus 原因分析の骨子

  全体を貫く根本原因 3 本

  R1: lease TTL が LLM 処理時間を想定していない（カテゴリ1 / 3件）
  TTL は「OS の UI が変化しない期間」として設計されているが、LLM
  の思考・ツール呼び出し生成時間は独立した軸。50件返却時ほど失効しやすいのは TTL が延びたのではなく LLM
  処理時間が縮んだから。次元違いの設計。

  R2: flat window model が modal / owner / 階層構造に非対応（カテゴリ 3・4・5・9 / 14件）
  Windows のウィンドウは owner/parent/modal の階層構造を持つが、MCP は hwnd + windowTitle の flat な識別子で動く。Save
  As の「親で read できるが touch は modal_blocking」「ダイアログ hwnd は uia_provider_failed」「V1 のリゾルバは
  zOrder=0 なのに見えない」はすべてここから派生。カテゴリ 4→3 の連鎖、5→9 の連鎖もここが起点。

  R3: provider capability の negative space がレスポンスに出ない（カテゴリ 2・7 / 5件）
  「terminal textbox は type 不可」「CDP なし・visual GPU 未発動」という情報が entity の属性として返ってこないため、LLM
  は「0件 = アプリが空」「role=textbox なら type できる」と誤推論する。できないことが見えない設計。
